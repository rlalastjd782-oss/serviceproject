# Records Style Dimensional UI

## 1. 목적

오늘탭과 기록탭의 UI가 서로 맞지 않아 앱 전체가 한 제품처럼 보이지 않는 문제를 해결한다. 기록탭의 비교적 안정적인 카드 구조, 간격, 활성 상태, 입체감을 기준으로 오늘탭을 포함한 전체 화면의 시각 체계를 통일한다.

이번 작업은 새 기능 추가가 아니라 기존 UI의 공통 디자인 방향을 재정의하는 변경 작업이다. 사용자는 "변경이 안 된다"고 느끼고 있으므로, 디자인/개발 단계에서는 실제 반영 여부를 DOM/HTML/CSS 계산값과 수동 브라우저 확인 권장 사항으로 검증해야 한다.

## 2. 사용자 시나리오

- 사용자는 오늘탭에서 운동 또는 식단을 기록하다가 기록탭으로 이동해도 같은 앱 안에 있다는 느낌을 받는다.
- 사용자는 오늘탭의 카드, 버튼, 입력 영역이 기록탭과 같은 깊이감과 간격 규칙을 따른다고 느낀다.
- 사용자는 주요 액션과 선택 상태가 평면적으로 묻히지 않고, 적당한 입체감으로 구분되길 기대한다.
- 사용자는 앱 전체가 너무 회색이거나 너무 납작한 화면이 아니라, 밝은 표면 위에 카드와 컨트롤이 또렷하게 떠 있는 느낌을 기대한다.
- 사용자는 변경 후 실제 화면에서 오늘탭 UI가 기록탭과 확실히 가까워졌는지 확인하고 싶어 한다.

## 3. 주요 화면/상태

- 오늘탭
  - 요약 모드
  - 운동 모드
  - 식단 모드
  - 날짜 선택, 모드 전환, 타이머, 입력 폼, 최근 기록, 요약 카드
- 기록탭
  - 기준 UI 레퍼런스
  - 일간 기록 카드, 검색/필터, 기간/정렬 컨트롤, 빈 상태
- 분석탭
  - 주간/월간/연간 요약 카드
  - PR, 운동별, 장비별, 부위별 분석 카드
- 더보기탭
  - 보조 기능 진입 카드
  - 캘린더, 장소, 식단 템플릿, 데이터 센터 등
- 공통 상태
  - 기본 카드
  - 활성/선택 카드
  - 입력 중
  - 저장 중/저장 완료
  - 빈 상태
  - 오류 상태
  - 모바일 좁은 화면

## 4. 기능 요구사항

- 기록탭을 전체 UI 기준점으로 삼는다.
  - 기록탭의 카드 간격, 카드 그림자, 모서리, 섹션 구분, 필터 스타일을 먼저 정리한다.
  - 오늘탭은 기록탭과 동일한 표면 규칙을 따르되, 운동/식단 입력 흐름에 필요한 우선순위는 유지한다.
- 전체 화면에 입체감을 부여한다.
  - 바깥 섹션 카드에는 명확하지만 과하지 않은 shadow, border, background 차이를 둔다.
  - 내부 row, chip, badge, input은 바깥 카드보다 낮은 깊이 단계로 표현한다.
  - 모든 요소가 같은 높이로 떠 보이는 과한 shadow는 피한다.
- 오늘탭과 기록탭의 불일치를 줄인다.
  - 오늘탭의 요약/운동/식단 카드가 기록탭 카드와 같은 시각 언어를 사용해야 한다.
  - 오늘탭만 지나치게 평면적이거나, 반대로 과하게 회색/흰색이 반복되는 상태를 피한다.
  - 타이머, 세트 입력, 식단 입력 같은 작업형 UI도 기록탭 카드 체계 안에서 자연스럽게 보여야 한다.
- 공통 UI 토큰을 정의한다.
  - page background
  - section card
  - elevated card
  - inset row
  - input/control
  - primary action
  - selected state
  - disabled/read-only state
  - empty state
- 활성 상태를 더 분명하게 만든다.
  - 오늘/기록/분석/더보기 상단 네비게이션의 현재 위치가 명확해야 한다.
  - 오늘탭 내부 요약/운동/식단 모드 선택이 기록탭의 필터/탭 활성 상태와 비슷한 강도로 보여야 한다.
  - 선택된 필터, 날짜, 정렬 상태는 주변 컨트롤보다 한 단계 떠 보이거나 대비가 높아야 한다.
- 모바일 사용성을 유지한다.
  - 입체감을 추가하더라도 화면이 답답해지지 않게 간격과 높이를 관리한다.
  - 카드 shadow 때문에 리스트가 뭉개지거나, 작은 화면에서 콘텐츠가 과하게 밀리지 않아야 한다.
  - 긴 한글 라벨과 숫자 지표가 버튼/카드 안에서 겹치지 않아야 한다.
- 실제 변경 검증을 요구한다.
  - 디자인/개발 완료 후 오늘탭과 기록탭을 나란히 비교할 수 있는 DOM/HTML/CSS 확인 기준을 남긴다.
  - "변경이 안 됨" 문제를 막기 위해 service worker cache, CSS import 순서, 최종 override 적용 여부를 확인한다.

## 5. 예외 케이스

- 이전 UI 1차 패스 QA가 아직 끝나지 않은 경우
  - 해당 작업의 QA 결과를 지우지 않는다.
  - 이번 작업은 별도 스펙과 handoff로 관리한다.
- 기록탭 스타일을 그대로 복사하면 작업형 UI가 불편해지는 경우
  - 오늘탭의 입력/타이머 흐름은 유지하고, 표면/깊이/간격 규칙만 맞춘다.
- 입체감을 과하게 적용하는 경우
  - 모든 작은 chip과 badge에 shadow를 주지 않는다.
  - 깊이는 2~3단계로 제한한다.
- 전체 회색화로 되돌아가는 경우
  - 최근 피드백상 all-gray 방향은 제외한다.
  - 밝은 카드와 적당한 그림자, 낮은 채도의 내부 표면을 조합한다.
- PWA 캐시 때문에 변경이 보이지 않는 경우
  - 개발 단계에서 `VERSION`, manifest, service worker cache 갱신 여부를 확인한다.
- 읽기 전용 사용자
  - 쓰기 액션은 비활성 상태로 보여도 카드/표면 규칙은 동일하게 유지한다.

## 6. 완료 조건

- 오늘탭과 기록탭을 나란히 봤을 때 카드 깊이, 간격, 활성 상태가 같은 디자인 체계로 보인다.
- 오늘탭의 요약/운동/식단 모드가 기록탭보다 납작하거나 이질적으로 보이지 않는다.
- 전체 UI에 적당한 입체감이 생기지만, 과한 shadow나 중첩 카드 느낌은 없다.
- 기록/분석/더보기 화면도 동일한 카드와 컨트롤 규칙을 따른다.
- 모바일 기준 DOM/HTML/CSS 검증에서 텍스트, 버튼, 필터, 카드가 겹치거나 화면 밖으로 넘치지 않는다.
- 변경 후 CSS 적용 순서, 버전, 주요 selector 계산값으로 "변경이 반영됨"을 확인할 수 있다.

## 7. 개발 참고사항

- 이번 스펙은 기존 `docs/specs/ui-refinement-pass.md`의 후속 변경 스펙이다.
- 이전 작업을 삭제하거나 되돌리지 말고, 기록탭 스타일 기준의 전체 UI 통일 작업으로 별도 처리한다.
- 기록탭 관련 CSS와 오늘탭 관련 CSS의 책임 범위를 먼저 확인한다.
- 마지막 override만 추가해 문제를 숨기기보다, 공통 표면 규칙이 여러 화면에 재사용되도록 정리하는 것이 바람직하다.
- 개발 단계에서는 CSS 적용 순서, service worker cache, manifest cache, `VERSION` 갱신 여부를 반드시 확인한다.
- 가정: 사용자가 말한 "기록탭처럼"은 기록탭의 카드형 표면, 간격, 활성 컨트롤, 적당한 입체감을 기준으로 전체 UI를 맞추자는 의미로 해석한다.
- 가정: 이번 작업의 최우선 비교 대상은 오늘탭과 기록탭이며, 분석/더보기는 공통 규칙 확산 범위다.

## 8. QA 체크포인트

- 오늘 요약, 오늘 운동, 오늘 식단, 기록 일간 화면을 같은 모바일 너비 기준의 DOM/HTML/CSS 조건으로 확인한다.
- 오늘탭과 기록탭의 카드 shadow, border, background, padding이 같은 계열인지 비교한다.
- 오늘탭 모드 전환과 기록탭 필터/탭 활성 상태의 강도가 비슷한지 확인한다.
- 운동 타이머와 입력 폼이 입체감은 있지만 과하게 떠 보이지 않는지 확인한다.
- 식단 입력과 식단 기록 카드가 기록탭 카드 체계와 맞는지 확인한다.
- 분석/더보기 카드가 오늘/기록과 다른 테마처럼 보이지 않는지 확인한다.
- PWA 새로고침 또는 캐시 갱신 후에도 변경된 CSS가 실제로 보이는지 확인한다.
- PNG/JPG/WebP 이미지와 스크린샷 파일은 만들지 않는다. 필요한 경우 사람에게 수동 브라우저 확인 항목만 남긴다.
