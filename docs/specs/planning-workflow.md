# Planning Workflow Spec

## 목적

이 문서는 앞으로 이 기획 스레드에서 접수되는 모든 신규 기능 요청을 기능별 작업 단위로 분리하고, 기획 산출물과 다음 역할 인계를 일관되게 관리하기 위한 운영 스펙이다.

## 적용 범위

- 이 프로젝트의 신규 기능 기획 요청
- 기존 기능 변경/개선 기획 요청
- 여러 요청이 한 번에 들어오는 경우의 작업 분리
- 기획 완료 후 디자인 담당자에게 전달하는 handoff 관리
- 역할별 진행 상태를 기록하는 `docs/agent-status.md` 관리

## 작업 시작 절차

1. 모든 기획 작업은 시작 전에 `docs/agent-status.md`를 먼저 확인한다.
2. 기획 행에 진행 중인 작업이 있으면 기존 산출물과 충돌하지 않게 이어서 정리한다.
3. 디자인, 개발, QA가 끝나지 않은 작업이 있으면 해당 작업의 스펙과 handoff를 덮어쓰지 않는다.
4. 한 번에 여러 요청이 들어오면 기능별로 분리한다.
5. 분리된 작업은 각각 별도 스펙 문서 또는 명확한 변경 섹션을 가진다.

## 신규 기능 처리 규칙

- 새 기능 요청은 `docs/specs/` 아래에 기능명 기준 소문자 kebab-case 파일로 작성한다.
- 스펙 문서에는 최소한 다음 내용을 포함한다.
  - 기능 목적
  - 사용자 문제
  - 핵심 사용자 흐름
  - 화면/상태 요구사항
  - 데이터/정책 요구사항
  - 예외/빈 상태/에러 상태
  - 우선순위와 범위 제외 항목
  - 다음 역할 전달사항

## 기존 기능 수정 처리 규칙

- 기존 스펙이 있으면 해당 스펙에 변경 이력을 추가한다.
- 기존 스펙이 없거나 변경 범위가 독립적이면 `docs/specs/{feature-name}-change-{date}.md` 형식의 변경 스펙을 만든다.
- 디자인/개발/QA가 아직 완료되지 않은 기존 작업은 보존하고, 새 요청은 별도 대기 작업으로 남긴다.

## 대기열 관리

- 여러 작업이 동시에 대기하면 `docs/agent-status.md`의 기획 행 또는 관련 스펙 문서에 대기 중 작업 목록을 남긴다.
- handoff 파일에는 다음 단계로 넘길 가장 우선순위 높은 작업 하나를 적는다.
- 후속 작업은 대기열로 남기고 기존 handoff의 대상 기능을 덮어쓰지 않는다.

## 기획 완료 절차

1. 해당 작업의 스펙 문서를 최신 상태로 저장한다.
2. `handoff/planning-to-design.md`를 최신 작업 기준으로 갱신한다.
3. `handoff/planning-to-design.md`의 상태를 `디자인 준비됨`으로 바꾼다.
4. `docs/agent-status.md`의 기획 행에 현재 작업, 산출물, 다음 액션을 갱신한다.
5. 다음 액션에는 디자인 담당자가 해야 할 일을 명확히 적는다.

## 현재 결정사항

- 이 기획 스레드는 앞으로 모든 신규 기능 요청을 계속 접수하는 단일 기획 창구로 사용한다.
- 요청은 매번 새 작업 단위로 처리한다.
- 기능별 산출물은 `docs/specs/`에 남긴다.
- 디자인 인계 산출물은 `handoff/planning-to-design.md`에 남긴다.
- 상태판은 `docs/agent-status.md`를 기준으로 한다.

## 다음 역할 전달사항

- 디자인 담당자는 새 기능 스펙을 받을 때 `handoff/planning-to-design.md`의 대상 기능을 우선 확인한다.
- 대기 중 작업이 있더라도 handoff에 적힌 우선순위 작업부터 처리한다.
- 디자인 완료 후에는 별도 handoff 규칙에 따라 `handoff/design-to-dev.md`와 `docs/agent-status.md`의 디자인 행을 갱신한다.
