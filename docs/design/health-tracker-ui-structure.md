# Health Tracker UI Structure

## 1. 화면 목적

헬스 트래커의 주요 화면은 사용자가 모바일 환경에서 운동, 식단, 신체 지표를 빠르게 기록하고 최근 상태를 확인하는 것을 목표로 한다. 현재 앱은 `오늘`, `기록`, `분석`, `식단`, `더보기`, `설정` 탭으로 구성되며, 기록 입력과 분석 확인을 같은 디자인 언어 안에서 반복 사용한다.

이번 문서는 `docs/specs`에 별도 기능 기획 문서가 없는 상태에서, 현재 프로젝트의 UI 구조와 재사용 가능한 컴포넌트 패턴을 개발자가 바로 따를 수 있도록 정리한 기준 문서다.

## 2. 사용자 흐름

1. 사용자는 상단 고정 헤더에서 앱 상태와 계정 정보를 확인한다.
2. 상단 탭 메뉴에서 `오늘`, `기록`, `분석`, `식단`, `더보기`, `설정` 중 목적 화면으로 이동한다.
3. `오늘` 화면에서는 날짜를 먼저 선택한 뒤 `운동`, `식단`, `요약` 모드 중 하나를 선택한다.
4. `운동` 모드에서는 운동 입력, 오늘 기록, 휴식 타이머, 루틴 진행, 회복 상태를 순서대로 확인한다.
5. `식단` 모드에서는 식단 입력, 오늘 식단 기록, 목표/도움말 영역을 확인한다.
6. `요약` 모드에서는 오늘의 핵심 수치, 상태 패널, 최근 기록을 훑어본다.
7. `분석` 화면에서는 하위 탭으로 주간, 월간, 연간, 운동, PR, 장비 분석을 전환한다.
8. `더보기` 화면에서는 운동 관리, 보조 도구, 분석/기록, 식단/데이터 도구를 카드 링크로 탐색한다.

## 3. 레이아웃 구조

앱은 모바일 우선 단일 컬럼 레이아웃을 기본으로 한다. `body`는 최대 폭 760px 중앙 정렬이며, 실제 콘텐츠는 `.content` 안에서 14px 좌우 여백과 하단 여백을 가진다.

공통 구조:

- `header.header`: 상단 고정 영역. 가운데 앱 타이틀, 우측 계정/버전 메타.
- `nav.tabs`: 상단 고정 탭. 각 탭은 동일 너비를 가지며 활성 탭은 파란 하단선으로 표시.
- `main.content`: 화면별 본문. 페이지는 `.section` 단위로 나뉜다.
- `nav.subnav`: 분석 등 2차 탐색에 사용. 가로 스크롤 가능한 칩형 탭.

오늘 화면 구조:

- `.date-row`: 이전/다음 날짜 이동과 날짜 입력.
- `.today-mode-actions`: 운동, 식단, 요약 모드 전환 버튼.
- `.workout-action-dock`: 운동 모드에서 입력, 기록, 휴식, 완료로 이동하는 sticky 빠른 이동 영역.
- `.section`: 요약 카드, 입력 폼, 기록 리스트, 상태 패널의 기본 컨테이너.
- `.summary-grid`, `.focus-grid`, `.today-task-grid`: 수치 요약과 다음 액션을 보여주는 카드 그리드.

더보기/도구형 화면 구조:

- 상단 현황 strip으로 핵심 수치를 먼저 보여준다.
- 기능 그룹별 `.section`을 나누고, 내부는 `.more-link-grid`와 `.more-link-card`로 구성한다.
- 관리/설정형 화면은 폼과 리스트를 같은 section 안에 배치하되, 추가/편집 영역은 `details`를 활용해 접을 수 있게 한다.

## 4. 주요 컴포넌트

- `section`: 화면의 기본 정보 단위. 8-10px radius, `var(--line)` border, 흰색 계열 panel, 가벼운 shadow를 사용한다.
- `section-header`: 좌측 제목/보조 설명, 우측 주요 액션을 배치한다. 제목은 짧게, 보조 설명은 상태나 기준을 설명한다.
- `btn-primary`: 현재 선택된 모드나 주요 실행 버튼. 파란색 계열을 사용한다.
- `btn-small`: 보조 액션, 이동, 필터 적용, 카드 안 액션에 사용한다.
- `btn-danger`: 삭제/위험 작업에만 사용한다.
- `tab-btn`: 최상위 내비게이션. 텍스트는 짧게 유지하고 active 상태를 반드시 제공한다.
- `subnav`: 분석처럼 같은 정보군 안에서 기간/종류를 전환할 때 사용한다.
- `summary-card`, `goal-card`, `today-task-card`: 핵심 수치, 목표 진행률, 다음 행동을 보여준다.
- `record-card`, `detail-row`, `mini-list-row`: 기록 목록과 세부 데이터를 표시한다.
- `empty`, `compact-empty`: 데이터가 없을 때 표시한다. 가능한 경우 다음 행동으로 이어지는 버튼을 같은 section 안에 배치한다.
- `compact-select`, `month-picker-form`, `date-picker-form`: 필터/기간 선택에 사용한다.

## 5. 상태별 UI

기본 상태:

- 각 화면은 사용자가 지금 할 수 있는 1차 행동을 상단 또는 첫 section 우측에 둔다.
- 숫자 데이터는 `font-variant-numeric: tabular-nums`가 적용되는 기존 숫자 스타일을 유지한다.
- 카드 안 텍스트는 긴 운동명, 장소명, 장비명이 들어와도 `overflow-wrap: anywhere` 계열 처리를 유지한다.

활성/선택 상태:

- 최상위 탭은 `.tab-btn.active`로 파란색 텍스트와 하단선을 사용한다.
- 모드 전환은 선택된 항목에 `btn-primary`, 나머지 항목에 `btn-small`을 적용한다.
- 완료된 목표/루틴은 기존 `.is-complete`, `.is-done` 패턴을 따른다.

로딩 상태:

- 서버 렌더링 화면은 기본적으로 즉시 표시되지만, 비동기 저장/오프라인 큐 상태는 `.offline-queue-status` 같은 고정 토스트로 노출한다.
- 저장 버튼은 제출 중 중복 클릭을 막고, 텍스트는 `저장 중` 또는 동일 길이의 짧은 문구로 바뀌어 레이아웃이 흔들리지 않게 한다.

빈 상태:

- 기록, 루틴, 식단, 검색 결과가 없으면 `.empty` 또는 `.compact-empty`를 사용한다.
- 빈 상태 문구는 이유와 다음 행동을 한 문장 안에서 명확히 제시한다.
- 입력 가능한 화면에서는 빈 상태 근처에 `추가`, `입력`, `오늘 기록` 같은 직접 행동 버튼을 둔다.

에러 상태:

- 폼 유효성 에러는 해당 입력 그룹 가까이에 짧은 문장으로 표시한다.
- 위험/실패 메시지는 `var(--danger)` 계열을 사용하되, section 전체를 붉게 만들지 않는다.
- 네트워크/오프라인 실패는 하단 토스트와 폼 주변 메시지를 함께 사용한다.

비활성/읽기 전용 상태:

- 방문자 모드는 기존 `.visitor-mode` 규칙처럼 post form, 편집 버튼, 내보내기 링크를 숨긴다.
- 비활성 카드에는 `.is-disabled`와 opacity 패턴을 사용한다.
- 비활성 버튼은 클릭 불가 상태와 시각적 opacity를 같이 제공한다.

## 6. 반응형 기준

- 기본 설계 폭은 모바일이며, 최대 760px 안에서 정보 밀도를 높인다.
- 2열 이상 그리드는 `repeat(auto-fit, minmax(...))` 또는 `minmax(0, 1fr)` 패턴을 사용한다.
- 날짜/필터/폼 액션은 좁은 화면에서 wrap되거나 단일 컬럼으로 내려가야 한다.
- sticky 요소는 `header`, `tabs`, `workout-action-dock` 순서로 겹침이 없어야 한다.
- 텍스트가 긴 카드와 버튼은 최소 높이를 고정하고, 버튼 안 텍스트가 넘치면 줄바꿈 또는 짧은 라벨로 조정한다.
- 데스크톱 폭에서도 마케팅형 히어로를 만들지 않고, 실제 작업 화면의 정보 구조를 유지한다.

## 7. 접근성 고려사항

- 날짜 이동, 기간 이동, 주요 nav에는 `aria-label`을 유지한다.
- 입력 필드는 보이는 label 또는 `aria-label`을 반드시 가진다.
- 색상만으로 상태를 전달하지 말고, active/complete/error 텍스트나 아이콘 의미를 함께 제공한다.
- 버튼과 링크는 최소 터치 높이 34-40px를 유지한다.
- sticky action dock이 화면을 가릴 수 있으므로 `html`의 `scroll-padding-top` 기준을 유지한다.
- `details/summary`를 사용하는 접이식 영역은 summary 문구만으로 내부 목적이 이해되어야 한다.
- 폼 에러와 저장 결과는 키보드/스크린리더 사용자도 인지할 수 있도록 입력 근처 또는 status 영역에 표시한다.

## 8. 개발 전달사항

- 새 화면은 `layouts/base.html`의 header, tabs, `.content` 구조를 그대로 사용한다.
- 새 공통 UI는 먼저 `health_tracker/templates/macros/ui.html` 매크로 재사용 가능성을 검토한다.
- 스타일은 기존 CSS 변수와 radius 기준을 따른다: `--panel`, `--panel-2`, `--line`, `--text`, `--strong`, `--muted`, `--blue`, `--green`, `--danger`, `--radius`.
- 신규 컴포넌트 radius는 기본 8px, 페이지 section은 8-10px 범위 안에서 맞춘다.
- 기능별 CSS가 필요하면 기존 분리 구조를 따라 `static/css/feature_pages.css`, `today.css`, `meal.css`, `records.css`, `analysis.css` 중 가장 가까운 파일에 제한적으로 추가한다.
- 새 기능 문서가 `docs/specs`에 추가되면 기능명 기준의 별도 디자인 문서를 `docs/design/{feature-name}.md`로 작성한다.
- 현재 템플릿 출력에서 일부 한글이 깨져 보이는 구간이 있다. 디자인 구현 전 파일 인코딩/렌더링 경로를 확인해 실제 사용자 화면의 문구 품질을 먼저 보정해야 한다.
- 버튼, 입력창, 상태 메시지, 로딩, 빈 상태, 에러 상태는 각 화면 설계에서 누락 없이 명시한다.

## 구현 요약

- 화면은 모바일 우선 단일 컬럼, 상단 고정 탭, section 기반 정보 묶음을 유지한다.
- 새 기능의 첫 화면은 요약보다 실제 입력/확인 업무를 먼저 배치한다.
- 카드와 그리드는 기존 8px radius, 밝은 panel, 절제된 border를 사용한다.
- 상태 UI는 active, loading, empty, error, disabled를 모두 설계하고 개발 태스크에 포함한다.
- `docs/specs`가 비어 있으므로 현재 문서는 프로젝트 공통 UI 기준으로 사용하고, 이후 기능 기획이 생기면 기능별 디자인 문서로 분리한다.
