# 기능 파이프라인

이 프로젝트(Health Tracker PWA)는 기획 → 디자인 → 개발 → QA → 최종검수 → 기획 의도검수 순서로 기능을 만든다. 각 단계는 `.claude/agents/`의 전용 서브에이전트가 맡고, 이 스킬은 그 순서를 조율하는 절차를 정의한다.

과거에는 별도 CLI 프로세스와 `handoff/*.md` 상태 파일로 단계를 넘겼지만, 지금은 한 세션 안에서 Agent 도구로 서브에이전트를 순서대로 호출하고 그 응답을 직접 읽고 다음 단계로 넘기면 된다. 별도 lock 파일이나 폴링이 필요 없다.

## 실행 순서

사용자 요청을 받으면 다음을 순서대로 진행한다. 각 단계 사이에는 이전 서브에이전트의 응답 요약(대상 기능명, 산출물 경로, 핵심 결정사항)을 다음 서브에이전트 프롬프트에 그대로 포함시킨다.

1. **기획 (planner)** — 사용자 요청을 그대로 전달해 `docs/specs/<feature-slug>.md`를 작성하게 한다. 기능명을 kebab-case slug로 정해 이후 모든 단계에서 동일하게 사용한다.
2. **디자인 (designer)** — 1번의 스펙 경로를 전달해 `docs/design/<feature-slug>.md`를 작성하게 한다.
3. **개발 (developer)** — 1, 2번 문서 경로를 전달해 구현과 테스트를 진행하게 한다.
4. **QA (qa-tester)** — 3번의 변경 요약을 전달해 `qa/reports/<feature-slug>-qa.md`를 작성하게 한다.
   - QA 판정이 배포 차단 수준이면 개발(developer)에게 이슈를 전달해 재작업시키고, QA를 다시 실행한다.
5. **최종검수 (final-reviewer)** — 1~4번 문서를 근거로 `qa/final-reviews/<feature-slug>-final-review.md`를 작성하게 한다.
   - 판정이 `보류`면 원인에 따라 개발 또는 QA 단계로 되돌리고, 해결 후 다시 최종검수를 실행한다.
   - 판정이 `승인` 또는 `조건부 승인`이면 다음 단계로 진행한다.
6. **기획 의도검수 (planner, 의도검수 모드)** — 5번 문서를 근거로 `docs/planning-final-reviews/<feature-slug>.md`를 작성하게 한다. planner 서브에이전트를 다시 호출하되, 프롬프트에 "이번에는 최종검수 이후 기획 의도검수 역할로 동작해줘"라고 명시한다.
   - 판정이 `기획 의도 보류`면 사용자에게 무엇이 기획 의도와 다른지 보고하고 다음 진행 방향을 확인한다.
   - 판정이 `기획 의도 승인`이면 Git 커밋 준비가 된 것이다.

## Git 반영

파이프라인이 끝까지 승인되어도 커밋/푸시는 자동으로 하지 않는다. 다음을 사용자에게 보여주고 명시적으로 확인받는다.

- `git status --short`로 변경된 파일 목록
- 스테이징할 파일(소스 코드만 — `docs/`, `qa/`, `handoff/`는 로컬 산출물이므로 커밋 대상에서 제외하되 `CHANGELOG.md`, `NOTES.md`는 공개 문서이므로 포함)
- 제안 커밋 메시지 (형식: `작업: <기능명> 최종검수 <승인|조건부 승인> 반영`)

사용자가 승인하면 커밋하고, push는 별도로 다시 확인한 뒤에만 진행한다.

## 참고

- 각 서브에이전트의 상세 역할/규칙은 `.claude/agents/planner.md`, `designer.md`, `developer.md`, `qa-tester.md`, `final-reviewer.md`에 있다.
- 단계를 생략하고 싶다는 요청(예: "QA는 생략하고 바로 최종검수")이 있으면 그대로 따르되, 생략한 단계를 최종 응답에 명확히 알린다.
- 이미 스펙/디자인 문서가 있는 기능을 다시 작업하는 경우, 처음부터 다시 쓰지 말고 기존 문서를 갱신하도록 각 서브에이전트에게 전달한다.
